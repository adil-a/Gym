# PR #1231 — Deep Review: *Allow the gym to run on Kubernetes*

**Verdict: `approve_with_minor_changes`**

> Generated by a 93-agent adversarial workflow (9 experts → dedupe → completeness critic → per-finding devil's-advocate → independent adjudication → consolidation). 50 raw findings → 32 canonical (+8 from the critic) → 40 adjudicated, 38 confirmed/partial.

## Executive summary

PR #1231 adds an optional, greenfield `k8s/` subtree that deploys NeMo Gym on Kubernetes/OpenShift via KubeRay, plus two CI workflows (image build + kind e2e). It is a thoughtful, security-conscious external contribution: full pod-level hardening (runAsNonRoot, readOnlyRootFilesystem, drop-ALL caps, seccomp RuntimeDefault), per-instance namePrefix+label isolation enabling multiple benchmarks in one namespace, an end-to-end kind smoke test, sensible graceful-shutdown rollout config, and arbitrary-UID/OpenShift support. Nothing here is a deploy-breaker, security hole, or data-loss bug; the happy path is exercised by CI and works.

After deduping the 38 input findings (F1–F32 plus C1–C8) and applying the adjudicator's verdicts, the substance reduces to a cluster of documentation-accuracy fixes and a few real engineering nits. Two documentation claims are materially false and should be corrected before merge: (1) the root README advertises "mTLS-authenticated distributed task execution" while zero TLS/mTLS is configured anywhere (all intra-cluster traffic is plaintext) — verified verbatim at README.md:311 with no TLS in k8s/; and (2) the README's "Pre-built images are available from GHCR" plus the verify-table "3/3" head-pod count are wrong/premature and will mislead early adopters. The only non-doc items worth acting on are CI hygiene (missing `concurrency` groups; helm operator unpinned vs README's 1.3.0), a coverage gap (cross-node Ray `.remote()` dispatch is never asserted by CI), and a probe/lifecycle coupling where the resources server's liveness probe bounces the whole Ray worker pod.

A large share of the original findings were downgraded or refuted on inspection: several cited line numbers from a NEWER base checkout than the PR branch (the `GET /` liveness endpoint did not exist when the PR was authored — F3), several "bugs" were defensible idiomatic choices (per-target gha cache scopes, gitignored secret.yaml, Burstable QoS, absent ServiceMonitor), and one (F24, bash on UBI10-minimal) was empirically refuted by pulling the image. Recommendation: approve with the documentation corrections and the cheap CI fixes; treat everything else as optional polish or follow-up.

## Confirmed findings


### MEDIUM

#### [F1] README claims "mTLS-authenticated distributed task execution" but no TLS/mTLS is configured anywhere  _(confidence 0.93)_

- **File:** `README.md:311 (also k8s/base/raycluster.yaml, k8s/README.md)`
- **Problem:** README.md:311 states the resources server is "embedded in the RayCluster for mTLS-authenticated distributed task execution." A scoped grep over k8s/ for mtls|tls|cert|RAY_USE_TLS|RAY_TLS_|mesh|istio|spiffe|networkpolicy returns ZERO matches (re-verified). KubeRay does not enable inter-node TLS by default; the RayCluster env blocks (head, gym-workers, gym-resources) set only RAY_TMPDIR/NVIDIA_VISIBLE_DEVICES/NEMO_GYM_*/*_HOST — no RAY_USE_TLS=1 and no cert mounts. Inter-service HTTP is also plaintext (server_utils.py builds f"http://..." URLs; probes hit plaintext /docs). So all Ray GCS/raylet/gRPC and all agent/model/resources/head HTTP traffic is unencrypted and unauthenticated, and the README claim is false.
- **Why it matters:** This is a security-sensitive, user-facing claim. An operator could be given a false sense of transport encryption for a worker group that executes untrusted benchmark/code-gen code. The defect itself is a single false adjective (no runtime/exploit impact — the plaintext posture exists independent of the sentence, and the actual k8s/ deployment guide does not repeat the claim and correctly notes port-forward-only access), which is why it is medium rather than high.
- **Fix:** Remove "mTLS-authenticated" from README.md:311 (e.g. "...embedded in the RayCluster for distributed task execution") and accurately state that intra-cluster traffic is plaintext HTTP/gRPC with isolation scoped to namespace + label selectors and no external exposure. If encrypted transport is actually intended, implement it (RAY_USE_TLS=1 + RAY_TLS_SERVER_CERT/KEY/CA mounted on every RayCluster container, TLS termination or a mesh sidecar on the FastAPI services) rather than asserting it.

#### [F9] Base ships :latest + imagePullPolicy: IfNotPresent on every workload; README's "images available" claim is premature  _(confidence 0.9)_

- **File:** `k8s/base/{deployment-agent,deployment-model,deployment-head,raycluster}.yaml, k8s/README.md:77, .github/workflows/build-k8s-images.yml`
- **Problem:** Every base workload pins :latest with imagePullPolicy: IfNotPresent and no overlay sets an images: transformer, so a node caching an older :latest silently runs stale code and Deployments do not roll on a re-push (unchanged pod template). The build workflow only pushes on non-PR events and tags :latest only on the default branch, so :latest does not exist until merge to main — yet README.md:77 states unconditionally "Pre-built images are available from GitHub Container Registry," guaranteeing ImagePullBackOff for pre-merge/fork deployers. NOTE: the original title's "non-reproducible / no way to pin without hand-editing" framing was empirically REFUTED — kustomize images: with digest: cleanly overrides base at the overlay level, and semver/sha tags exist post-merge.
- **Why it matters:** Unsafe defaults plus an affirmatively misleading docs claim that produces ImagePullBackOff for the most likely early adopters is a real DX/operational defect, though not a deploy-breaker for the documented merge-to-main flow.
- **Fix:** Add a README caveat that GHCR images publish only after merge to main (and that GHCR packages default to private until visibility is set), so pre-merge/fork deployers must build from source. Document that production should pin a version/digest and note IfNotPresent's no-rollout behavior. Add a digest: example to the README images:-override section (currently only newName/newTag is shown). Optionally pin Dockerfile base images by @sha256.

#### [F15] Cross-node Ray task dispatch is never exercised by CI (only the no-Ray overlay is tested)  _(confidence 0.9)_

- **File:** `k8s/overlays/example-single-tool-call/kustomization.yaml, .github/workflows/k8s-integration-test.yml`
- **Problem:** CI deploys only the example_single_tool_call overlay (which has no ray.remote calls). So a Ray-using resources server dispatching .remote() from the num_cpus=0 gym-resources node onto a gym-worker, and autoscaler scale-up, are not asserted by any CI step. NOTE: the original "Ray is never exercised" framing is overstated — the CI resources server DOES run initialize_ray() (NEMO_GYM_SKIP_RAY_INIT is absent from the resources worker; configmap sets ray_head_node_address: "auto") and joins the live cluster, and the novel resources-as-Ray-worker postStart launch IS validated by the reward==1 smoke test.
- **Why it matters:** The PR's central value (running a distributed Ray dispatch topology on k8s) has an untested slice. A regression in cross-node dispatch would not be caught. This is a coverage gap, not an asserted defect — the 0-CPU dispatch design and autoscaler RBAC were independently judged sound.
- **Fix:** Add a CI step that deploys a Ray-using overlay (e.g. code-gen, whose app.py issues check_correctness_remote.remote()) and asserts a task completes, proving dispatch onto gym-workers end-to-end. Drop the "assert the autoscaler scales 1→4" idea as infeasible on a single-node kind runner (gym-workers request cpu:2/mem:4Gi each); at most document autoscaler scale-up as unverified by CI, and consider minReplicas:0 for non-Ray overlays.

#### [F12] New CI workflows have no concurrency cancellation and trigger on broad source paths  _(confidence 0.9)_

- **File:** `.github/workflows/build-k8s-images.yml, .github/workflows/k8s-integration-test.yml`
- **Problem:** Neither new workflow defines a concurrency group, so rapid pushes queue duplicate full runs (4-image matrix build + kind e2e) with no cancel-in-progress. Both fire on broad paths including resources_servers/** — the dominant PR activity — so essentially every benchmark PR pays for the kind run. NOTE: PRs build-only (no GHCR push) and use GHA layer caching, so the real cost is duplicate queued runs, not a from-scratch 4-image build each time.
- **Why it matters:** Avoidable runner/wall-clock cost on every benchmark PR; diverges from the repo's dominant convention (unit-tests.yml and publish-fern-docs.yml use cancel-in-progress: true for PR-triggered runs). Cheap to fix.
- **Fix:** Add `concurrency: { group: ${{ github.workflow }}-${{ github.event.pull_request.number || github.ref }}, cancel-in-progress: true }` to both workflows (matches the dominant PR-triggered convention). Do NOT narrow the e2e trigger to k8s/** only — nemo_gym/server_utils.py is baked into the images and on the critical path; instead consider dropping responses_api_agents/** and responses_api_models/** from the e2e path filter since the mock-model e2e exercises neither.

#### [F4] Resources-server liveness probe on :9080 restarts the entire Ray worker pod, not just the wedged gym child  _(confidence 0.82)_

- **File:** `k8s/base/raycluster.yaml`
- **Problem:** In the gym-resources worker group, PID 1 is the Ray worker (ray start --block); the NeMo Gym resources server is only a detached postStart background child on :9080. The livenessProbe targets /docs:9080 — the child — so when it fails (~150s) the kubelet restarts the whole container, killing the raylet. A crashed child has no in-place supervisor, so recovery is the heavyweight container bounce. The fix is cheaper than implied: server_utils.py's initialize_ray() connects to Ray as a CLIENT via ray.init(address=...), so the resources server need not BE a Ray worker.
- **Why it matters:** Couples the lifecycle of the probed child to the raylet. A wedged-but-alive server bounces the Ray node instead of restarting the child in place. NOTE: harm is bounded — gym-resources runs num-cpus=0 (loses no scheduled compute), autoscaling tolerates worker churn, and the 150s threshold rules out transient stalls — which is why this is medium, not high.
- **Fix:** Decouple the server from the raylet. Preferred: run the resources server as its own Deployment (mirroring deployment-agent/model) and let it connect to Ray as a client via the existing initialize_ray() path, giving it its own PID 1 so liveness restarts only the server. Minimal: drop the livenessProbe on :9080 (keep readiness, which de-routes the dead endpoint without bouncing the raylet) and add an in-place supervisor for the postStart child. Add a comment noting the probe currently health-checks a detached background child.

#### [F27] Troubleshooting curl expands $POLICY_BASE_URL/$POLICY_API_KEY on the local client, not in the pod  _(confidence 0.95)_

- **File:** `k8s/README.md:369-370`

#### [F31] Resources pod's /tmp/nemo-gym-server.log shares the 4Gi ray-tmp emptyDir with the Ray session dir and grows unbounded  _(confidence 0.9)_

- **File:** `k8s/base/raycluster.yaml (also deployment-head/agent/model tmp sizing, which is fine)`
- **Problem:** On the gym-resources pod, the postStart hook appends to /tmp/nemo-gym-server.log with no rotation, RAY_TMPDIR=/tmp points Ray's session dir at the same /tmp, and /tmp is the ray-tmp emptyDir with sizeLimit:4Gi. - **Fix:** Redirect the postStart entrypoint output to container stdout/stderr (drop `>> /tmp/nemo-gym-server.log 2>&1`, letting kubectl logs capture it) or rotate/cap the file so it cannot accrete into the gym-resources 4Gi budget. Do not frame as a spill+log collision filling 4Gi.

#### [F8] Editable install + readOnlyRootFilesystem causes silent first-party __pycache__ write failures; PYTHONDONTWRITEBYTECODE unset  _(confidence 0.88)_

- **File:** `k8s/Dockerfile, k8s/base/{raycluster,deployment-*}.yaml`
- **Problem:** Images use `pip install -e .` so nemo_gym source lives on the read-only root; at runtime readOnlyRootFilesystem:true with only /tmp writable means CPython cannot write __pycache__ next to the first-party source. CPython degrades gracefully (no crash), so cold starts recompile first-party bytecode without caching, and the read-only-root behavior is an accident of CPython's fallback rather than declared intent. NOTE: pip precompiles the dependency wheels into the image, so only the small first-party nemo_gym tree recompiles — the original 'no caching across the board' framing is overstated.
- **Why it matters:** Performance/hygiene: a tiny one-time first-party recompile per cold start, amortized over a long-lived pod. Cheap, correct fix.
- **Fix:** Set `ENV PYTHONPYCACHEPREFIX=/tmp/pycache` (so caching succeeds in the only writable dir) or `ENV PYTHONDONTWRITEBYTECODE=1` in the Dockerfile, OR build and install a wheel (with `python -m compileall`) so bytecode is baked at build time. Reframe to first-party-only recompile.

## Refuted / dismissed by the adversarial pass

- **F24 — Head entrypoint.sh #!/bin/bash may not run on python-312-minimal** — REFUTED empirically. Pulling the exact base image registry.access.redhat.com/ubi10/python-312-minimal shows /bin/bash and /usr/bin/bash are both present (1.4MB), and CI builds+rolls out the head pod (its #!/bin/bash entrypoint must exec for the test to pass). Moreover the proposed #!/bin/sh fix is unsafe: entrypoint.sh:2 uses `whoami &> /dev/null`, and bash vs dash diverge on that line, which would flip the arbitrary-UID branch. The author's #!/bin/bash is correct. Optional only: `microdnf install -y bash` to make the dependency explicit.
- **C7 — No PodDisruptionBudget / PriorityClass / HPA availability primitives** — REFUTED on substance. 'No HPA' is false and the recommendation is harmful — KubeRay in-tree autoscaling is already enabled (enableInTreeAutoscaling:true, gym-workers minReplicas:1/maxReplicas:4); a CPU-target HPA would fight the Ray autoscaler. QoS is intentionally Burstable (memory limits set, CPU limits omitted), a best practice, and orthogonal to PDBs. A PDB on the single-replica Deployments/worker groups cannot protect work (minAvailable:1 blocks drains; maxUnavailable:1 is a no-op). Graceful-shutdown levers (preStop sleep, terminationGracePeriodSeconds, maxUnavailable:0) are already present. Only a nit-level residual (the single gym-resources worker loses in-flight work on a voluntary drain) remains, mitigated by re-runnability; the proposed fixes don't fit single-replica batch components.
- **C2 — No PVC; rollout outputs on emptyDir lost on restart** — Persistence/data-loss premise refuted. The deployed pods are stateless HTTP services that never write rollout artifacts to disk — rollouts are produced by ng_collect_rollouts on the operator's machine via port-forward to a client-side path. The emptyDirs hold only ephemeral Ray scratch and the relaunchable server log. readOnlyRootFilesystem + writable /tmp is intentional hardening, and a PVC would conflict with the autoscaling worker group (RWO single-attach). Only a minor observability note survives (the /tmp server log not appearing in kubectl logs and lost on restart), already partly covered by F5 and the README; 'Add PVC' is rejected.
- **F32 — No CPU limits (Burstable QoS) and RayCluster does not honor Deployment rollingUpdate semantics** — Net non-issue. The CPU-request-without-limit Burstable QoS is a deliberate, defensible choice (avoids CFS throttling), and the manifests ship no LimitRange/Namespace so the LimitRange-injection caveat is hypothetical. The cross-API rolling-update difference is not a defect — the RayCluster CRD has no rollingUpdate field, so the manifests make no false zero-downtime promise. The only durable nugget is an optional one-line doc note that RayCluster pods are reconciled by KubeRay rather than Deployment maxUnavailable:0 semantics. No action required for merge.
- **F29 — Architecture diagram / data-flow text is code_gen-specific** — Confirmed but lowest-severity nit, folded into the docs-accuracy cluster rather than tracked separately. k8s/README.md:26's 'agent extracts code → Ray workers execute unit tests' is code_gen-specific and inaccurate for example_single_tool_call, but the harm is mild (the architecture box already shows both cg- and estc- instances and the Verify table documents the worker pods). Worth a small wording fix (generic '/verify scores the task' plus a code_gen-specific note); not a blocker.
- **F30 — Two similarly-named head services (gym-head-svc vs KubeRay's <cluster>-head-svc)** — Confirmed-but-rejected-remedy nit. There is no name collision; the names differ by the meaningful -ray- infix, are on distinct ports (11000 vs 8265), and live in distinct README sections, so confusion is unlikely. The proposed rename has real blast radius (head Service selector label, deployment-head labels, overlay replacements) for zero functional gain and should NOT be done. The only worthwhile action is the optional one-line doc note that cg-gym-ray-head-svc is auto-created by KubeRay — non-blocking.

## Strengths

- Comprehensive, consistent pod-level security hardening on every container: runAsNonRoot, readOnlyRootFilesystem, allowPrivilegeEscalation:false, capabilities drop ALL, seccompProfile RuntimeDefault — a genuinely above-average baseline for a community k8s contribution.
- Real multi-instance isolation via per-overlay namePrefix + app.kubernetes.io/instance labels with includeSelectors, plus kustomize replacements and CRD name-reference propagation, correctly preventing resource-name collisions and Service cross-selection so multiple benchmarks can co-exist in one namespace (network-level segmentation is the documented caveat, F16).
- An actual end-to-end kind integration test (helm-install KubeRay → build+load 4 images → roll out agent/model/head + RayCluster → port-forward → assert a full cross-pod round-trip) that exercises the genuinely novel mechanism of this PR: running the resources server inside a Ray worker group via a postStart raylet-socket-wait launch.
- Thoughtful graceful-shutdown and rollout configuration on the Deployments (maxUnavailable:0/maxSurge:1, minReadySeconds, preStop sleep, terminationGracePeriodSeconds) and correct use of KubeRay in-tree autoscaling for the compute worker group.
- Deliberate arbitrary-UID / OpenShift support (chgrp/chmod g=u on /opt/app-root, ENV HOME, an openshift overlay Component) and a sensible client-proxy model that keeps GPU/inference out of cluster, reusing the existing initialize_ray() client-connect path.
- Clean greenfield scoping: the entire feature lives in an optional k8s/ subtree and two new workflows, touches the core library minimally (the NEMO_GYM_SKIP_RAY_INIT short-circuit), gitignores secrets correctly with a committed .example template, and is well-documented overall — the issues found are predominantly documentation-accuracy fixes rather than design flaws.
- The author demonstrably understood the constraints: explicit WARNING comments on the brittle env-index patches, a documented kubectl-exec path for the file-based server log, a non-fail-fast guard that no-ops rather than errors on read-only root, and a port choice (9080) made specifically to avoid Ray's metrics port — these show care even where the chosen tradeoffs are debatable.
