# Kubernetes for a SLURM user — guided by PR #1231

## What this PR does

PR #1231 adds a `k8s/` directory that lets you run NeMo Gym benchmarks on any Kubernetes cluster, with KubeRay providing the distributed Ray execution. It is almost purely additive: everything lives under `k8s/` plus a single one-line edit to `nemo_gym/server_utils.py`. The design splits the four NeMo Gym roles across cluster workloads: the stateless servers (simple_agent, vllm_model proxy, and a config-only HeadServer) run as standard Deployments behind ClusterIP Services, while the resources server and the code-execution workers run inside a KubeRay RayCluster. The clever/novel bit is that the resources server is not its own Deployment — it is launched by a `postStart` hook as a background child inside a num-cpus:0 Ray worker Pod, so it can both serve HTTP /verify AND dispatch Ray `.remote()` tasks onto the real num-cpus:2 worker group (which autoscales 1->4). Configuration is wired with Kustomize: a platform-agnostic `base/` plus per-benchmark overlays (code-gen `cg-`, example `estc-`) that set a `namePrefix`, an instance label, image overrides, and `replacements` to rewrite Service-DNS env vars — so multiple benchmarks coexist in one namespace. It targets an external OpenAI-compatible LLM endpoint (no in-cluster GPU/vLLM, no training token-id capture), exposes nothing externally (port-forward only), hardens every container (non-root, read-only root FS, dropped capabilities, seccomp), and ships two new CI workflows: one that builds the four images and one that runs a kind-based end-to-end smoke test of the example overlay. The findings show the *deployment is functionally healthy*; the defects are concentrated in (a) docs overclaims (mTLS, 'images available', 3/3 head, ng_status, network isolation), (b) probe/lifecycle coupling on the resources Pod, and (c) CI coverage gaps (Ray dispatch and the tool path are not asserted).

## Concept map: Kubernetes ↔ SLURM

### Pod
**≈ SLURM:** A single allocated task slot that wraps one or more co-scheduled processes — closest to one `srun` step running on one node, but the unit of scheduling itself.

A Pod is the smallest deployable unit in K8s: one or more containers that always run together on the SAME node, sharing a network namespace (same localhost, same Pod IP) and shared volumes. In SLURM, the scheduler hands you a node allocation and you launch processes onto it; in K8s, the Pod IS the allocation-plus-process bundle, and the scheduler (kube-scheduler) decides which node it lands on. Where the analogy breaks: a SLURM step is ephemeral and you control its launch directly; a Pod is a declared object the cluster keeps trying to place and (often) keeps restarting. In THIS PR the `gym-resources` worker Pod is the most unusual: PID 1 is `ray start --block` (the raylet), and the actual NeMo Gym resources FastAPI server is a *second* process launched by a `postStart` hook into the background — so one Pod runs two logically separate services. That co-location is exactly why finding F4 bites: a liveness probe failure on the FastAPI child restarts the whole Pod (raylet included).

### Deployment (vs a long-running job)
**≈ SLURM:** A self-resubmitting batch job with `--requeue` on steroids: you declare 'I want N copies of this service running forever,' and a controller keeps resubmitting/replacing them to maintain that count, including rolling restarts on config change.

A Deployment is a controller object that manages a long-running, stateless service. You give it a Pod template and a replica count; its job is to make reality match that declaration continuously. Unlike `sbatch` (fire-and-forget: you submit, it runs to completion, it's done), a Deployment never 'completes' — if a Pod dies, it's recreated; if you change the Pod template (new image, new env), it performs a rolling update: spin up new Pods, wait for them Ready, then tear down old ones. THIS PR uses Deployments for the three stateless HTTP servers: `gym-agent`, `gym-model`, `gym-head`. The Ray nodes are NOT Deployments — they're managed by the RayCluster CRD instead. Where it breaks: SLURM jobs are imperative and time-bounded; Deployments are declarative and unbounded. The RollingUpdate strategy here (`maxSurge:1, maxUnavailable:0`) means zero-downtime swaps — there's no clean SLURM analog because SLURM has no notion of 'keep the old one serving until the new one is healthy.'

### ReplicaSet / replicas
**≈ SLURM:** The `--array=0-N` (job array) count, but enforced continuously rather than submitted once. `replicas: 3` is like guaranteeing exactly 3 array tasks are alive at all times.

A Deployment doesn't manage Pods directly — it creates a ReplicaSet, and the ReplicaSet's only job is 'keep exactly N matching Pods alive.' Each rolling update creates a NEW ReplicaSet (new Pod template) and scales the old one to zero. You rarely touch ReplicaSets directly; you reason in terms of the Deployment's `replicas` field. In THIS PR every Deployment sets `replicas: 1` (single agent, single model, single head) — it's a reference deployment, not a scaled-out production one. The Ray worker groups use a different but analogous concept (`replicas`/`minReplicas`/`maxReplicas` in the RayCluster spec). Where the analogy breaks: a SLURM array task that fails just shows up as failed in `sacct`; a ReplicaSet actively *recreates* the failed Pod to restore the count, so 'how many ran' and 'how many are running' are different questions in K8s in a way they aren't in a one-shot array.

### Service + ClusterIP + DNS
**≈ SLURM:** A stable hostname/alias for a service that may move between nodes — like having a fixed DNS name that always points at 'whatever node is currently running the parameter server,' instead of hardcoding the compute node's hostname you got from `squeue`.

Pods are mortal and get new IPs every restart, so you never address a Pod directly. A Service is a stable virtual IP (the ClusterIP) plus a DNS name that load-balances across all Pods matching its label selector. KubeDNS/CoreDNS resolves `gym-resources-svc` (or fully-qualified `gym-resources-svc.gym.svc.cluster.local`) to that virtual IP. THIS PR is built entirely on this: the agent reaches the model via `MODEL_HOST=gym-model-svc`, the model reaches the agent via `gym-agent-svc`, and the resources worker reaches both via Service names (see the env blocks). The Service's selector (`app.kubernetes.io/name: gym-resources`) is how it finds its backing Pods. Where it breaks vs SLURM: there's no SLURM equivalent of a built-in virtual-IP load balancer with DNS — you'd normally pass the node list around manually or use a side-channel. Finding F1 matters here: all this Service traffic is plaintext HTTP, despite the README's 'mTLS' claim. Finding F17 matters too: the *base* Service selectors are name-only, so if you ever applied base alongside an overlay in one namespace, the unprefixed Service could cross-select the overlay's Pods.

### Namespace
**≈ SLURM:** A SLURM partition combined with an account/project scope — a named, quota-able, access-controlled slice of the cluster where your objects live and are addressed.

A Namespace is a virtual cluster-within-the-cluster: a naming and isolation boundary. Object names must be unique *within* a namespace (two namespaces can each have a `gym-config`). DNS is namespace-aware (`svc.gym.svc.cluster.local`), RBAC and resource quotas are typically scoped per-namespace. THIS PR puts everything in namespace `gym` and runs *multiple benchmark instances in that one namespace*, relying on `namePrefix` (cg-, estc-) rather than separate namespaces to keep them apart. Where it breaks: a SLURM partition is mainly about node sets and scheduling policy; a Namespace is purely a logical/API boundary and does NOT by itself isolate the network. That distinction is the heart of finding F16 — there's no NetworkPolicy, so being in the same namespace gives naming isolation but NOT network isolation; a malicious code-gen Pod could still reach another instance's Ray GCS or the credentialed model service over the flat Pod network.

### ConfigMap
**≈ SLURM:** A shared, version-controlled config file (or a block of `export VAR=...` lines) that you stage onto the node and your job reads — except it's a first-class cluster object, not a file on a shared filesystem.

A ConfigMap stores non-secret configuration as key/value data, injected into Pods either as env vars or mounted as files. THIS PR's `gym-config` ConfigMap holds the entire Hydra config as one `config.yaml` key, and every server reads it via `NEMO_GYM_CONFIG_DICT` (env var sourced from `configMapKeyRef`). The config uses `${oc.env:...}` interpolation so the same ConfigMap works across servers — each server picks its slice with `NEMO_GYM_CONFIG_PATH` (agent uses `simple_agent_instance`, model uses `policy_model`). Where it breaks vs SLURM: editing a ConfigMap does NOT automatically restart consumers — Pods that read it as env vars keep the old values until recreated (similar to how exporting a new value doesn't change an already-running job, but here the config object updated silently and nothing rolled). Finding F19 is relevant: when a server boots from `NEMO_GYM_CONFIG_DICT`, the config-ref validation that `ng_run` normally does is skipped, so a dangling `resources_instance` reference only fails at request time, not boot time.

### Secret
**≈ SLURM:** The same idea as a ConfigMap but for credentials — like a locked-down file with 0600 perms holding your API token that the job sources, except managed by the cluster API with (optional) at-rest encryption and tighter RBAC.

A Secret is base64-encoded key/value data intended for sensitive values, with stricter access controls than a ConfigMap (and can be encrypted at rest). THIS PR's `gym-secrets` holds `POLICY_BASE_URL`, `POLICY_API_KEY`, `POLICY_MODEL_NAME` — the external LLM credentials — and is injected via `envFrom: secretRef` into the agent, model, and resources containers. The secret.yaml is created per-overlay from secret.yaml.example and is intentionally NOT checked in. Where it breaks vs SLURM: base64 is encoding, not encryption — a Secret is only 'secret' relative to RBAC and at-rest config, not inherently. And finding F27 is a concrete gotcha: the README's troubleshooting `curl` command references `$POLICY_API_KEY` without an in-pod shell wrapper, so your LOCAL shell expands it to empty before kubectl runs — the secret lives only inside the Pod's env, so the documented diagnostic always fails.

### ServiceAccount / RBAC
**≈ SLURM:** Your SLURM user/account identity plus the sacctmgr-defined permissions on what you may submit and where — but here it's an identity assigned to the *workload itself*, governing what the Pod's code may ask the cluster's control plane to do.

A ServiceAccount is the identity a Pod runs as when it talks to the Kubernetes API server. RBAC (Roles + RoleBindings) grants that identity specific verbs on specific resource types. THIS PR uses a `gym-sa` ServiceAccount on all Pods. Crucially, the stateless Deployments set `automountServiceAccountToken: false` — they never call the K8s API, so they get no token (least privilege). The Ray Pods DO keep the token because the KubeRay autoscaler needs to call the API to create/delete worker Pods when scaling. Where it breaks vs SLURM: a SLURM job's identity governs *scheduling and accounting*; a ServiceAccount governs *control-plane API calls made by the running workload* — a category SLURM doesn't really have, since SLURM jobs don't reshape the cluster as they run. The autoscaler RBAC here was independently judged sound in the findings.

### kubectl (vs sbatch / srun / squeue)
**≈ SLURM:** One CLI that fuses sbatch (submit/apply), srun (exec into a running task), squeue/sacct (get/describe/list state), and scancel (delete) — all driven by declarative YAML rather than imperative submission.

`kubectl` is the client to the cluster's API server. Rough mapping: `kubectl apply -f` ≈ `sbatch` (but declarative — you describe desired state and the cluster reconciles); `kubectl get pods` / `kubectl describe` ≈ `squeue` / `sacct`; `kubectl exec -it pod -- bash` ≈ `srun --pty` / sshing into the allocated node; `kubectl delete` ≈ `scancel`; `kubectl logs` ≈ tailing the job's stdout file. THIS PR's whole workflow is `kustomize build overlays/code-gen | kubectl apply -f -`. Where it breaks: `sbatch` returns a job ID for a thing that will end; `kubectl apply` registers desired state for things that persist and self-heal. Finding F5 is a kubectl-logs gotcha: the resources FastAPI server logs to `/tmp/nemo-gym-server.log`, not stdout, so `kubectl logs` on that Pod shows raylet output instead of the server — you must `kubectl exec ... cat` the file (and the CI 'Resources Logs' step is mislabeled because of this).

### liveness / readiness / startup probes
**≈ SLURM:** Health checks SLURM mostly lacks: a recurring 'is this still alive?' watchdog (liveness ≈ a node health-check script that drains/reboots a bad node), a 'is it ready for traffic yet?' gate (readiness — no SLURM analog, since SLURM has no load balancer to gate), and a 'still booting, don't judge it yet' grace window (startup ≈ a prolog that must finish before the step counts as up).

Three distinct kubelet-run checks. Startup probe: runs first, gives a slow-booting container a long grace window before the other probes apply (here, resources allows ~330s: failureThreshold 60 × period 5s + initialDelay 30s). Readiness probe: when it fails, the Pod is removed from its Service's endpoints — traffic is de-routed but the Pod is NOT killed. Liveness probe: when it fails, the kubelet RESTARTS the container. THIS PR points all three at `path: /docs` (the FastAPI Swagger page). Finding F3: `/docs` returns 200 as soon as uvicorn accepts a socket, so it's a shallow signal that can keep a wedged-but-listening server 'healthy.' Finding F4 (the sharpest one): on the resources Pod, liveness on `:9080` targets the FastAPI child, but PID 1 is the raylet — so a liveness failure restarts the WHOLE container, killing the raylet, not just the wedged child. Readiness would have de-routed it without the bounce.

### requests vs limits (vs SLURM resource requests)
**≈ SLURM:** `requests` ≈ `--mem`/`--cpus-per-task` used for *scheduling/placement* (the guaranteed reservation); `limits` ≈ a cgroup hard cap that SLURM can also enforce — but K8s lets request < limit, creating a 'burstable' middle zone SLURM's single number doesn't have.

`requests` are what the scheduler uses to place a Pod: it only lands on a node with that much CPU/memory free, and that amount is guaranteed. `limits` are the hard ceiling enforced at runtime: exceed the memory limit and the container is OOM-killed; exceed the CPU limit and it's throttled (not killed). When request < limit, the Pod is 'Burstable' — it can use spare capacity up to the limit but is the first to be evicted under node pressure. THIS PR: the `gym-workers` request cpu:2/mem:4Gi and limit cpu:4/mem:8Gi (room to burst for test execution); `gym-resources` requests cpu:500m but runs `num-cpus:0` in Ray so it schedules no Ray compute. Where it breaks vs SLURM: SLURM typically reserves exactly what you asked and that's both the guarantee and the cap; K8s deliberately separates 'what's guaranteed' from 'what's allowed,' which enables overcommit but also surprise OOM-kills and evictions that have no clean SLURM analog. Finding F15 notes the gym-workers' cpu:2 request is why CI can't verify autoscaler scale-up on a single-node kind runner — four such workers won't fit.

### Kustomize base / overlay / namePrefix
**≈ SLURM:** Job-script templating: a canonical `template.sbatch` (base) plus per-experiment override fragments (overlays) and a prefix on every job/output name to keep concurrent experiments from clobbering each other — but done by structured YAML merging, not envsubst/sed.

Kustomize layers YAML without templating placeholders. The `base/` holds platform-agnostic manifests; an `overlay/` references the base and applies patches, a `namePrefix`, extra labels, image overrides, and `replacements`. THIS PR's `code-gen` overlay sets `namePrefix: cg-` (so `gym-agent` becomes `cg-gym-agent`), injects `app.kubernetes.io/instance: cg` into selectors and templates, patches the resources worker's entrypoint, and uses `replacements` to rewrite Service-name env vars to the prefixed names. This is what lets two benchmarks coexist in one namespace. Where it breaks vs sed-templating: Kustomize merges by structured paths, so it's more robust than text substitution — BUT finding F7 shows the limit: the RayCluster CRD has no merge keys, so the env patches address vars by absolute *array index* (`env/2`, `env/4`). Reorder the base env list and you silently mis-patch. Finding F9/F28 also live here: base pins `:latest` and no overlay sets an `images:` digest, so re-pushes don't trigger rollouts and pre-merge deployers hit ImagePullBackOff.

### KubeRay RayCluster + head/worker groups + autoscaling
**≈ SLURM:** Standing up a Ray cluster the way you would under SLURM with a head-node job + worker-node jobs sharing an address — but here a Kubernetes *operator* (the KubeRay controller) owns the lifecycle and can grow/shrink the worker pool automatically, like an elastic `salloc` that adds nodes on demand.

RayCluster is a Custom Resource (a CRD KubeRay installs); the KubeRay operator watches it and creates the head Pod and worker-group Pods. THIS PR declares one head group (`num-cpus:0`, so it schedules no tasks — it's GCS/dashboard/coordination only) and two worker groups: `gym-workers` (num-cpus:2, the actual code-execution muscle, autoscaling 1→4) and `gym-resources` (num-cpus:0, max 1 — it hosts the FastAPI resources server and dispatches `.remote()` tasks but runs none itself). `enableInTreeAutoscaling: true` makes KubeRay inject an autoscaler sidecar into the head Pod (which is why the head is 2/2, not 3/3 — finding F26). Where it breaks vs a hand-rolled SLURM Ray cluster: under SLURM you'd statically allocate worker nodes for the job's lifetime; KubeRay's autoscaler adds/removes worker Pods dynamically based on pending Ray tasks, calling the K8s API to do it. Finding F15: CI never deploys a Ray-*using* overlay, so cross-node `.remote()` dispatch from gym-resources onto gym-workers is exercised at runtime but not asserted by any test.

### securityContext / non-root / readOnlyRootFilesystem
**≈ SLURM:** Running your job as an unprivileged user on a node where `/` is mounted read-only and only your scratch dir is writable — like a hardened compute node where you can't write outside `$TMPDIR` and can't escalate to root.

A securityContext hardens the container: `runAsNonRoot: true` refuses to start as UID 0; `allowPrivilegeEscalation: false` blocks setuid escalation; `capabilities.drop: [ALL]` strips Linux capabilities; `readOnlyRootFilesystem: true` makes the entire container filesystem read-only except explicitly mounted writable volumes; `seccompProfile: RuntimeDefault` applies the default syscall filter. THIS PR applies all of these to every container and mounts an `emptyDir` at `/tmp` as the only writable path. This matters because the gym-workers run *untrusted candidate code* from code-gen benchmarks. Where it breaks vs SLURM: SLURM relies on the node OS and your user account for isolation; here it's per-container kernel-level confinement. Two findings interact with read-only root: F8 (CPython can't write `__pycache__` next to the editable-installed source, so it silently recompiles bytecode every cold start — harmless but wasteful) and F21 (the `/etc/passwd` arbitrary-UID append in entrypoint.sh is a no-op because root is read-only, so it never fires).

### lifecycle.postStart
**≈ SLURM:** A task prolog / a backgrounded setup step that runs right after the main process launches — like an `srun` step that, once the node is up, forks a secondary daemon before the real work, except it runs in the same container as PID 1.

`lifecycle.postStart` is a hook the kubelet runs immediately after a container's main process starts. It runs in the same container. THIS PR uses it for the cleverest and riskiest piece of the design: in the `gym-resources` worker, PID 1 is `ray start --block` (the raylet), and the postStart hook backgrounds a loop that waits for the raylet socket to appear, then `exec`s the NeMo Gym resources FastAPI server. This is how one Pod becomes both a Ray node AND the resources HTTP server. Where it breaks / risks: postStart runs concurrently with the main process and its failure handling is weak. Finding F6: the socket-wait `until` loop is unbounded and backgrounded, so postStart returns immediately and the Pod reports Running even if the socket never appears (the ~330s startup probe is the only backstop). Finding F4 again: because the FastAPI server is a detached child with no in-place supervisor, if it wedges, recovery is the heavyweight Pod restart, not a child restart.

### OpenShift arbitrary-UID
**≈ SLURM:** Running on a cluster that, for security, assigns your job a RANDOM unprivileged UID at launch (not your usual uid) — so any script that assumes a known username or writes to a fixed home dir must cope with 'I don't know who I am.'

OpenShift's default SCC runs containers as a random high UID not present in the image's /etc/passwd. Software that calls getpwuid (to find a username/home) can fail. The standard mitigation is to make the relevant dirs group-writable (`chmod g=u`) and group-owned by root (GID 0), since the random UID always has GID 0, and to set HOME explicitly. THIS PR ships an `openshift` overlay (a Kustomize Component that sets `imagePullPolicy: Always`) and the image's entrypoint.sh attempts an /etc/passwd nss fallback. Where it breaks: finding F21 shows the /etc/passwd append is inert under `readOnlyRootFilesystem` (can't write /etc/passwd), so the load-bearing OpenShift hardening is actually the chgrp/chmod on /opt/app-root plus ENV HOME — the passwd branch never fires in the shipped config. There's no SLURM analog for 'the platform gives my process an identity that doesn't exist in any user database.'

### port-forward (vs ssh tunnel to a compute node)
**≈ SLURM:** Almost exactly an `ssh -L 8080:localhost:8080 compute-node` tunnel to reach a service bound on an allocated node — but it tunnels through the K8s API server to a Pod/Service, with no SSH daemon on the target and no inbound firewall opening.

`kubectl port-forward svc/cg-gym-agent-svc 8080:8080` opens a local socket that forwards through the API server to the target Pod/Service. THIS PR deliberately exposes NOTHING externally (no LoadBalancer/Ingress); the README is explicit that all access — the agent API, the Ray dashboard on 8265, the head server on 11000 — is via port-forward only. This is the documented, correct security posture. Where it breaks vs ssh tunnel: there's no shell or sshd involved, it's an API-server-mediated stream, and it dies when your kubectl process exits; it's strictly for ad-hoc/dev access, not production traffic. This is also why finding F18 matters: you port-forward to the head server to run CLI tools like `ng_collect_rollouts`, but `ng_status` against the cluster always reports 'no servers found' because the standalone HeadServer never registers instances — so the analogy of 'tunnel in and inspect' partly fails for that one command.

## Architecture walkthrough (one request through the code-gen deployment)

Here is one full request traced through the code-gen overlay (namePrefix `cg-`), naming every Pod and Service it touches. Think of it as one `srun` of the eval pipeline, except every hop is an HTTP call to a Service DNS name resolved by CoreDNS, and the 'compute' at the end is a Ray `.remote()` dispatch.

PARTICIPANTS (all in namespace `gym`):
- Deployment `cg-gym-agent` -> Pod `cg-gym-agent-*` (simple_agent FastAPI on :8080), fronted by Service `cg-gym-agent-svc`.
- Deployment `cg-gym-model` -> Pod `cg-gym-model-*` (vllm_model proxy on :8080), fronted by Service `cg-gym-model-svc`.
- RayCluster `cg-gym-ray` -> head Pod `cg-gym-ray-head-*` (ray-head + autoscaler sidecar = 2/2; GCS:6379, dashboard:8265, client:10001, num-cpus:0); worker group `gym-workers` Pods `cg-gym-ray-gym-workers-worker-*` (num-cpus:2, the code executors, autoscale 1->4); worker group `gym-resources` Pod `cg-gym-ray-gym-resources-worker-*` (num-cpus:0; PID1=raylet, postStart child=resources FastAPI on :9080), fronted by Service `cg-gym-resources-svc`.
- Deployment `cg-gym-head` -> Pod `cg-gym-head-*` (HeadServer config endpoint :11000), fronted by Service `cg-gym-head-svc`. The head is OUT of the request path — it only serves config to local CLI tools via port-forward.

STEP 0 (entry). A developer runs `kubectl port-forward svc/cg-gym-agent-svc 8080:8080` (the ssh-tunnel analog) and POSTs a task to `http://localhost:8080/run`. The forward lands on Service `cg-gym-agent-svc`, which load-balances (here, trivially, replicas:1) to the `cg-gym-agent-*` Pod. (In a rollout collection, the local `ng_collect_rollouts` does this same /run call.)

STEP 1 (agent -> model). simple_agent in `cg-gym-agent-*` runs its harness loop. It first seeds session state against the resources server, then calls the model. It resolves `MODEL_HOST=cg-gym-model-svc` (an env var that Kustomize `replacements` rewrote from the base `gym-model-svc` to the prefixed name) via CoreDNS to the model Service's ClusterIP, and POSTs to the model server's responses/chat endpoint on :8080. This is plaintext HTTP (finding F1 — there is no mTLS despite the README's claim).

STEP 2 (model -> external LLM). The `cg-gym-model-*` Pod is a vllm_model *proxy*, not a GPU server. It reads `POLICY_BASE_URL` / `POLICY_API_KEY` / `POLICY_MODEL_NAME` from the `cg-gym-secrets` Secret (via envFrom) and makes an outbound HTTPS call to the external OpenAI-compatible endpoint. `return_token_id_information: false` and `NVIDIA_VISIBLE_DEVICES: void` confirm this is eval-against-hosted-API only, no in-cluster inference or training token-id capture (finding C3). The LLM's completion flows back: external LLM -> model Pod -> agent Pod.

STEP 3 (agent extracts). Back in `cg-gym-agent-*`, simple_agent parses the model's response and extracts the candidate code / final answer from the trajectory.

STEP 4 (agent -> resources /verify). The agent resolves `RESOURCES_HOST=cg-gym-resources-svc` via CoreDNS and POSTs the candidate to the resources server's verify endpoint on :9080. Note what this Service points at: NOT a Deployment, but the `gym-resources` *Ray worker Pod*. Inside that Pod the request is served by the FastAPI process that the postStart hook launched after the raylet socket appeared (the F4/F6 mechanism). Note the agent Pod itself has `NEMO_GYM_SKIP_RAY_INIT=1` and `automountServiceAccountToken: false` — it never joins Ray and never calls the K8s API.

STEP 5 (resources -> Ray dispatch). The resources server's verify() for code-gen calls Ray. It is connected to the cluster as a Ray *client* (ray_head_node_address: "auto", joined to `cg-gym-ray`), but with num-cpus:0 it schedules nothing locally — so its `check_correctness_remote.remote()` calls dispatch the unit-test execution onto the `gym-workers` Pods (num-cpus:2 each), the only group with schedulable CPU. If demand exceeds capacity, the KubeRay autoscaler (sidecar in the head Pod) calls the K8s API to add gym-workers Pods up to maxReplicas:4. This cross-node dispatch (resources Pod -> head GCS -> gym-workers Pod) is the PR's central value and the slice CI does not assert (finding F15).

STEP 6 (run tests -> reward). The gym-workers Pod executes the candidate's unit tests inside its hardened container (non-root, readOnlyRootFilesystem, /tmp emptyDir, all caps dropped — because this is untrusted code). Pass/fail comes back through Ray to the resources server, which computes a reward (0.0 or 1.0) and returns it to the agent. The agent returns the completed trajectory + reward out through `/run` to the caller.

CONTROL-PLANE SIDE NOTES that shape the path: CoreDNS resolves every `*-svc` name; the Secret supplies the only credentials in the system; the ConfigMap (`cg-gym-config`) supplies the Hydra config each server slices via `NEMO_GYM_CONFIG_PATH`; the autoscaler is the only component using the `gym-sa` token (the stateless Deployments disabled it). And every hop in steps 1, 4, 5, 6 is plaintext HTTP or plaintext Ray gRPC over a flat Pod network with no NetworkPolicy (findings F1 + F16).

## Key risks in plain language

### F1 — README claims 'mTLS-authenticated' but nothing in k8s/ configures TLS or mTLS.

Think of it like a README telling you all inter-node traffic in your Ray cluster goes over an encrypted, mutually-authenticated channel — and then you find every node is actually talking plain unencrypted TCP, no certs anywhere. The deployment isn't broken (it never needed TLS to function), but an operator reads 'mTLS' and assumes the wire is protected, which it is not: all agent/model/resources HTTP and all Ray GCS/gRPC traffic is plaintext. It's a single false adjective, but a security-sensitive one for a system that runs untrusted benchmark code. Fix is a one-line doc correction (or actually implement RAY_USE_TLS + mounted certs + a mesh/termination layer if encryption is really intended).

### F4 — the resources Pod's liveness probe restarts the whole Ray worker, not just the wedged server.

On that one Pod, two processes share the same allocation: the Ray node (PID 1) and the resources HTTP server (a backgrounded child). The liveness watchdog only checks the HTTP child. In SLURM terms, imagine a node-health script that, when your secondary daemon hangs, doesn't restart just that daemon — it reboots the entire node, killing the Ray worker process that was perfectly fine. So a wedged-but-alive server triggers a heavyweight container bounce instead of a cheap child restart. Harm is bounded (that Pod is num-cpus:0 so no scheduled compute is lost, and the 150s threshold rules out blips), but the lifecycle coupling is the wrong shape. The clean fix is to run the resources server as its own Deployment that joins Ray as a *client*, giving it its own PID 1.

### F9 / F28 — base pins ':latest' with IfNotPresent, and the README says pre-built images 'are available' when they aren't yet.

Two problems that bite the earliest users. First, ':latest' + 'IfNotPresent' is like running a job script that says 'use whatever build is already cached on the node, don't re-fetch' — so a node holding a stale ':latest' silently runs old code, and re-pushing ':latest' won't trigger a rolling update because the Pod template text didn't change. Second, the GHCR images are only published after this PR merges to main; if you're on a fork or reviewing the PR branch and follow the docs literally, every Pod gets stuck in ImagePullBackOff (the K8s equivalent of a job that can never start because the container image doesn't exist). Fix: a README caveat about the merge-timing and GHCR private-by-default, plus guidance to pin a digest/version and build-from-source pre-merge.

### F16 — no NetworkPolicy, so 'complete isolation' overclaims for untrusted-code workloads.

The PR isolates benchmark instances by *naming* (cg-, estc- prefixes and instance labels), which prevents name collisions and Service cross-selection. But naming isolation is not network isolation. In SLURM terms: putting two projects in the same partition keeps their job names distinct, but it doesn't firewall their processes from each other on the network. Here the Pod network is flat and there's no NetworkPolicy, so a malicious code-gen Pod (running arbitrary candidate code) could reach another instance's Ray GCS/dashboard or the credentialed model/head Services. The README's phrase 'complete isolation — no data bleed' reads broader than the naming-scoped reality. Fix is primarily to scope that claim honestly and optionally ship an opt-in default-deny NetworkPolicy (which also needs a policy-enforcing CNI).

### F15 — CI never exercises cross-node Ray dispatch (it only tests the no-Ray example overlay).

The whole point of the PR is the distributed topology: the resources server dispatching `.remote()` work onto separate worker Pods, with autoscaling. But CI only deploys the example-single-tool-call overlay, which makes no Ray `.remote()` calls. So the resources server does join the live Ray cluster in CI, but the actual fan-out-and-execute-on-a-worker path is run at runtime yet never *asserted* by a test. In SLURM terms, it's like having an integration test that confirms your head node boots and your worker nodes register, but never actually submits a step that runs on a worker — so a regression that breaks worker dispatch would sail through green CI. Fix: add a CI step that deploys a Ray-using overlay (code-gen) and asserts a dispatched task completes. (Asserting autoscaler 1->4 is infeasible on a single-node kind runner, so that's documented-as-unverified rather than tested.)

### F5 — the resources server logs to a file, so 'kubectl logs' shows the wrong thing.

The most useful debugging output — the resources FastAPI server's stdout — is redirected to /tmp/nemo-gym-server.log inside the Pod. But since the raylet is PID 1, `kubectl logs` on that Pod streams the raylet's output instead, and the CI failure-diagnostics step labeled 'Resources Logs' actually prints raylet noise. In SLURM terms, it's like your job's real stderr getting written to a scratch file while `sacct`/the slurm log shows the wrapper script's output — you go looking in the obvious place and find the wrong stream. The README does document the correct retrieval (`kubectl exec ... cat` the file), and probes make a truly broken server fail loudly, so it's low-severity — but it's a real debugging-ergonomics trap. Fix: have CI exec-and-cat the file, or tee the server log to the Pod's stdout (with rotation so it can't fill the 4Gi /tmp emptyDir, per F31).

The behavior SLURM's single number doesn't capture is the gap between them: when request < limit the Pod is 'Burstable' — it may use spare capacity up to the limit but is the first evicted under node pressure. In this PR, gym-workers request cpu:2 but may burst to limit cpu:4 for test execution.
