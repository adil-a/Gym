## (a) How OpenEnv does external agents

OpenEnv is fundamentally an **environment-server framework**, not a verification-layer framework. The README frames it as "an e2e framework for creating, deploying and using isolated execution environments for agentic RL training, built using Gymnasium style simple APIs" (`README.md:3`). The canonical contract is a per-session **WebSocket** (`/ws`) speaking Gymnasium `reset()`/`step()`/`state()` messages (`src/openenv/core/env_client.py:152`, `src/openenv/core/env_server/http_server.py:1309`); reward lives *inside* the environment's `step()` return (`StepResult.reward`, `src/openenv/core/client_types.py:24`). There is no first-class `/verify` HTTP endpoint — "verification" is a Python method, `ResourceSession.verify()`, on the orchestration side (`src/openenv/core/harness/__init__.py:131`).

External agents plug in through the **experimental harness layer** (RFC 005, `src/openenv/core/harness/__init__.py:7`), which sits *outside* the env server and has two paths. (1) **White-box** (`MCPHarnessAdapter.run_white_box`, line 489): the trainer owns model sampling via a `ModelStep` callback and OpenEnv runs the MCP tool-calling loop, collecting `prompt_ids/completion_ids/logprobs` per turn (`ModelStepResult`, line 71) for TRL/GRPO (`build_harness_rollout_func`, line 631). (2) **Black-box** (`CLIHarnessAdapter.run_black_box`, line 599): an opaque external CLI agent (Codex, and by symmetry Claude Code/Cursor) is shelled out as a subprocess; OpenEnv re-exposes the env's tools to it over an **MCP JSON-RPC bridge** (`SessionMCPBridge`, line 398) wrapped in a tiny stdlib HTTP server on `/mcp` (`examples/browsergym_harness_eval_common.py:399`), and the agent is pointed at it via `codex mcp add <env> --url ...` (`examples/browsergym_codex_eval.py:147`). The env can also serve MCP natively in **production mode** via `MCPEnvironment` + a FastAPI `POST /mcp` JSON-RPC route (`src/openenv/core/env_server/mcp_environment.py:110`, `http_server.py:1279`). A separate `transparent_proxy` mode (opencode env) intercepts the black-box agent's upstream LLM calls and writes `(token_ids, per_token_logps)` to a trace for GRPO (`envs/opencode_env/harness.py:17`).

Net: OpenEnv solves "bring-your-own external agentic client over MCP" and "this env runs RL training" very well, but it is an **environment/tool-serving** model, not a verification-API model. Adopting "its approach" in NeMo-Gym means adopting the harness/MCP-bridge *pattern* (re-expose a resources server as MCP tools + a subprocess CLI agent runner), not dropping in its code.

## (c) PROS of adopting this approach in NeMo-Gym

- **Bring-your-own agentic client is solved cleanly (R13/R14).** The `SessionMCPBridge` → stdlib HTTP `/mcp` → `codex mcp add --url` pattern is a small, copyable recipe to let Claude Code / Cursor / Codex drive a NeMo-Gym resources server as MCP tools without importing Gym (`harness/__init__.py:398`, `examples/browsergym_harness_eval_common.py:399`). Directly matches the Siemens "Fuse Agent: agentic client → MCP Tools + Skills → product servers" slide.
- **A skill-install mechanism exists (R14, partial).** `openenv skills add --claude/--codex/--cursor/--opencode` writes a `SKILL.md` into each client's skills dir (`skills.py:51,139`). **Caveat:** the only skill it ships is the static `openenv-cli` skill describing the `openenv` CLI itself (`skills.py:18-46`) — it is *not* the per-product "publish framework for agent skills" the Siemens slide calls for (which wants per-product Solido skills, custom-skill publishing, auto-loading, and skill collections). It's a reusable *plumbing* pattern (where to drop SKILL.md per client), not a skills publishing framework.
- **Reward-provenance discipline (R7).** `_resolve_env_reward` hard-fails if the orchestrator tries to invent a reward outside the env (`harness/__init__.py:213,236`) — a good invariant for trustworthy RLVR that NeMo-Gym could borrow regardless of routing.
- **Clean incremental ramp (R12).** The black-box (eval) → `transparent_proxy`/white-box (RL with logprobs) progression is explicit and lets an external team start with zero Gym code and deepen later — the exact spectrum #1396 asks for.
- **Config simplicity (R8).** Plain constructor args + `typer` CLI; nothing like Hydra. A useful contrast point for the #1396 "no OmegaConf learning curve" ask.
- **Runtime conformance check exists (R4).** `openenv validate --url` already emits a versioned criteria report — a template for the "fast-path validation tool" Gym wants.
- **Token-level RL from an opaque CLI is demonstrated (R7).** The `transparent_proxy` interception trick (`envs/opencode_env/harness.py:15-20`) shows how to get GRPO-grade per-turn `(messages, completion_tokens, per_token_logps)` written to `proxy_trace.jsonl` even when the agent is a black box — a genuinely useful pattern for Gym's training pipeline.

## (d) CONS

- **Different center of gravity: env-server, not verification-API.** OpenEnv's contract is WebSocket Gym `step/reset/state` with reward-in-env (`env_client.py`, `client_types.py:24`). NeMo-Gym's contract is `/run` → `/verify` → `BaseVerifyResponse(reward)`. There is no HTTP `/verify`; "verify" is a Python method requiring a `ResourceSession` wrapper (`harness/__init__.py:131`) — so R2's "hit /verify without wrapping the loop" is **not** what OpenEnv offers. Adopting it wholesale would re-platform Gym onto Gymnasium semantics.
- **Concurrency model is the opposite of R11.** No backpressure/capacity signaling; over-cap sessions are *rejected*, the worker pool is fixed at 32 threads, the MCP bridge is a stdlib `ThreadingHTTPServer`, and `CollectRunner` is serial (`http_server.py:228,338`, `harness/collect.py:283`). Scale comes from one-container-per-env horizontal scale-out, not 4k–65k concurrent in-process `/run`. This is the single biggest mismatch with Gym's stated requirement.
- **No model-server hosting / discovery / auth (R6).** OpenEnv never exposes the policy model; the agent brings its own (`llm_client.py:159`). Gym's "external harness reaches `/v1/responses` with URL discovery + auth" has no analog here.
- **No auth anywhere on env endpoints (R10/R15).** No Bearer/API-key/RBAC in `http_server.py`; cross-infra retry/backoff is also absent on the client. Multi-tenant / access-controlled data lake (R15) is entirely unaddressed.
- **Harness layer is explicitly experimental (RFC 005, pre-stable).** `harness/__init__.py:7` warns it "lives outside the stable `openenv.core` package surface while RFC 005 is still under review." Building Gym's external-agent strategy on it means tracking a moving target.
- **Black-box path is eval-only by default.** `examples/browsergym_codex_eval.py:1` states it is "eval-only... not suitable for white-box RL training or any setup that needs token IDs"; token-level training for opaque agents requires the more invasive `transparent_proxy` sandbox machinery.
- **No custom-output-format abstraction (R3).** Outputs are env-specific observation dataclasses or opaque text; there's no pluggable response-format layer.

## (e) Engineering-disruption rating: HIGH

Rationale: Adopting OpenEnv's *architecture* (not just borrowing the MCP-bridge pattern) means moving NeMo-Gym from its `/run`→`/v1/responses`↔`/<tool>`→`/verify` HTTP-microservice flow onto a WebSocket Gymnasium `reset/step/state` env-server contract with reward-in-env and Python-method verification (`env_client.py`, `harness/__init__.py:131`). That displaces the SimpleAgent loop, the `/verify` HTTP contract, the model-server tier (which OpenEnv doesn't host), and the concurrency story (Gym needs 4k–65k concurrent `/run` + backpressure; OpenEnv caps and rejects, and collects serially — `http_server.py:338`, `harness/collect.py:283`). The harness is also pre-stable (RFC 005). The high-value pieces (MCP bridge for BYO clients, the `skills` command, reward-provenance check, the `transparent_proxy` logprob trick) are *patterns* worth porting into Gym's existing flow at LOW-to-MEDIUM cost — but "adopt OpenEnv's external-agent approach" as the route is HIGH disruption.

## (f) FIT VERDICT

OpenEnv is the right **inspiration** but the wrong **wholesale route** for #1396. It nails the Siemens BYO-agentic-client surface (R13: any MCP-capable CLI drives the env over an MCP JSON-RPC bridge) and the incremental ramp (R12), and its MCP-bridge, the skill-install plumbing, reward-provenance guard, and `transparent_proxy` logprob capture are concrete patterns NeMo-Gym should lift into its existing `/run`→`/verify` flow. The **MCP-tool** half of R14 is fully there; the **Agent-Skills** half is only a thin scaffold (one static `openenv-cli` skill, no per-product skills / publishing / auto-discovery — so R14 is *partial*, not a full match for the Siemens skills vision). But OpenEnv is an environment-server framework (WebSocket Gym `step/reset`, reward-in-env, no HTTP `/verify`, no hosted model server, cap-and-reject concurrency, no auth/data-lake), so it does not satisfy Gym's verification-API-centric, high-concurrency, model-server-discovery, or multi-tenant data-lake requirements (R2/R6/R11/R15). Recommendation: borrow the MCP-bridge + skill-install plumbing + logprob-proxy patterns into Gym's current architecture rather than re-platforming onto OpenEnv. Net fit: **inspiration, not wholesale adoption** — strong on R12/R13 and the MCP half of R14; weak/absent on R2/R6/R11/R15 (the requirements that matter most for a verification-API, high-concurrency, multi-tenant Gym).
